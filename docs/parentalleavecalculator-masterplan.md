## 1. STRATEGIC THESIS

The Singapore parental leave SERP is owned almost entirely by government sites (MOM DR 81, MSF DR 73, ProFamilyLeave DR 73) and a few HR SaaS blogs (QuickHR, Info-Tech). **Not a single private-sector utility tool exists.** Every government page answers the *policy* question but provides zero interactivity — no calculators, no pay estimators, no leave planners, no shareable output.

`parentalleavecalculator.com` wins by being the most useful, interactive, and shareable destination for every Singapore parent trying to understand their leave entitlements. The domain name is exact-match for "parental leave calculator" — a globally searched term with 600+ monthly global volume where Singapore has no dedicated player.

**Total addressable keyword cluster (confirmed via Ahrefs MCP):** ~18,000–22,000 monthly searches in Singapore alone, with traffic potential exceeding 40,000+ when all long-tail programmatic pages are included.

## 8. TECHNICAL SEO ARCHITECTURE

### Next.js 14 Implementation Rules

**CRITICAL — SSR for all calculator pages:**
```
// WRONG — kills indexability
'use client'
export default function MaternityCalculator() {...}

// RIGHT — server shell + client calculator component
// app/maternity-leave/calculator/page.tsx
export const metadata = {
  title: 'Maternity Leave Calculator Singapore 2026 | Free Tool',
  description: 'Calculate your maternity leave entitlement...',
  alternates: { canonical: 'https://parentalleavecalculator.com/maternity-leave/calculator' }
}

export default function Page() {
  return (
    <main>
      <h1>Maternity Leave Calculator Singapore 2026</h1>
      <p>Calculate your exact entitlement...</p> {/* Static content above fold */}
      <MaternityCalculatorClient /> {/* 'use client' component below */}
    </main>
  )
}
```

### URL Structure Rules
- All lowercase, hyphen-separated
- No trailing slashes
- Canonical tag on every page pointing to itself
- Dynamic pages use generateMetadata() not static metadata
- Programmatic pages pre-generated at build time (generateStaticParams)

### Schema Markup (per page type)
- Calculator pages: `WebApplication` + `FAQPage`
- Guide pages: `Article` + `FAQPage` + `BreadcrumbList`
- Homepage: `WebSite` + `SiteLinksSearchBox`
- Comparison pages: `Table` (structured comparison)

### Sitemap Architecture
```
/sitemap.xml → auto-generated
/sitemap-calculators.xml → all /calculator paths
/sitemap-guides.xml → all /guides paths
/sitemap-programmatic.xml → all dynamic paths
```

### Core Web Vitals Targets
- LCP: < 2.0s (calculator widgets lazy-loaded below fold)
- CLS: 0.0 (fixed-height calculator container to prevent layout shift)
- INP: < 100ms (client-side calculation, no API calls for basic results)

## 9. SUPABASE SCHEMA

```sql
-- Shareable results storage
CREATE TABLE leave_calculations (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  share_code VARCHAR(12) UNIQUE NOT NULL,  -- short sharable code
  leave_type VARCHAR(50) NOT NULL,          -- 'maternity', 'paternity', etc.
  inputs JSONB NOT NULL,                    -- all calculator inputs
  results JSONB NOT NULL,                   -- calculated outputs
  created_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ DEFAULT NOW() + INTERVAL '90 days'
);

-- Policy data (updated when MOM changes rates)
CREATE TABLE leave_policies (
  id SERIAL PRIMARY KEY,
  leave_type VARCHAR(50) NOT NULL,
  effective_date DATE NOT NULL,
  policy_data JSONB NOT NULL,              -- entitlement rules
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Anonymous usage analytics (no PII)
CREATE TABLE calculation_events (
  id SERIAL PRIMARY KEY,
  leave_type VARCHAR(50),
  employment_type VARCHAR(50),
  child_order INT,
  has_salary BOOLEAN,
  result_shared BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Email subscribers (for annual policy update)
CREATE TABLE subscribers (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  leave_types TEXT[],                       -- which leave types they care about
  created_at TIMESTAMPTZ DEFAULT NOW(),
  confirmed BOOLEAN DEFAULT FALSE
);
```

## 10. CONTENT DIFFERENTIATION STRATEGY

### Why Government Sites Can't Beat You
1. MOM/MSF cannot run interactive calculators (policy, not product)
2. Government pages are static — no personalisation
3. Cannot generate shareable outputs
4. Cannot A/B test or iterate quickly
5. Zero social sharing features
6. No email capture capability
7. Cannot show "what changed from last year" comparisons

### Your Unfair Advantages
1. **Speed of update:** When MOM announces leave changes (April 2025 update, potential 2026 updates), you update the same day. Government sites take weeks.
2. **Interactive outputs:** The shareable result card is impossible for a government site
3. **Plain-English explanations:** MOM uses legislative language. You explain it in normal words.
4. **Scenario modelling:** "What if my baby is born early? What if I resign?" — government sites don't cover edge cases. You do.
5. **Mobile-first:** Government sites are notoriously poor on mobile. Your Next.js app will be perfect.

### Content Tone Guidelines
- Address the reader as "you" (not "employees" or "workers")
- Lead with the answer, then the explanation
- Use Singapore English naturally (e.g., "you will be entitled to")
- Include concrete dollar examples (salary: $5,000 → you get $X/month during leave)
- Always link to the MOM/MSF source for credibility
